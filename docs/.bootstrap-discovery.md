# Bootstrap Discovery Report — CardTrader Deal Scanner ("Card // Broker")

> **Project state: greenfield.** No implementation exists yet — only the PRD
> (`cardtrader-deal-scanner-PRD.md`), a high-fidelity design handoff
> (`design_handoff_deal_scanner_dashboard/`, React-in-Babel prototype, **not** to be
> ported as-is), and this report. All standards/conventions below are derived from the
> PRD's stated stack (§5), suggested repo structure (§14), and the design handoff —
> not from existing source, because none exists.

> **ARCHITECTURE DECISION (overrides PRD §5/§10/§12 frontend delivery):** The dashboard
> ships as a **Tauri desktop app**, not a browser SPA served from the Worker.
> Chosen architecture = **Tauri client + cloud backend**:
> - **Backend is unchanged from the PRD** — Cloudflare Worker (Hono + D1 + hourly cron
>   scan + Telegram). It stays always-on and scans regardless of whether the desktop app
>   is running. The Worker becomes **API-only** (no static-asset hosting of the dashboard).
> - **Frontend** = React + Vite + TypeScript SPA (recreating the design handoff) running
>   inside a **Tauri webview** with a **Rust host**. It calls the same cloud `/api/*` routes
>   over HTTPS via TanStack Query.
> - **Auth changes:** browser-based Cloudflare Access (PRD §12) is replaced for the desktop
>   client by a **Cloudflare Access service token or shared bearer secret** stored securely
>   on-device (Tauri store/stronghold). No public unauthenticated D1 routes either way.
> - New concerns vs PRD: Tauri host config, IPC/commands (minimal), secure on-device
>   credential storage, "open in system browser" for Buy links, desktop packaging +
>   code-signing + auto-update.

## Conventions observed
Derived from PRD + design handoff (no real source yet):
- **File naming:** `camelCase.ts` for backend modules (`dealEngine.ts`, `client.ts`), per PRD §14. React components `PascalCase.tsx` (TBD — confirm at scaffold).
- **Money:** integer **cents** everywhere; never store/compute floats. Format only at the display edge (PRD §6, README).
- **API shape:** JSON over Hono; resources under `/api/*`; PATCH for partial updates; `:id/reset` sub-route to null an override.
- **Inheritance pattern:** per-ticket override columns are `NULL` → fall back to `config` default at scan time (§9a). UI distinguishes `inherit · {default}` vs `override ✕`.
- **Import style:** ES modules, TypeScript. (Confirm ordering at scaffold.)
- **Error handling:** external-API failures are non-fatal per-blueprint (log + skip); whole-run errors recorded in `scan_runs.error`; 401 aborts the run cleanly + alerts once.
- **Prototype anti-patterns to AVOID** (README): no Babel-in-HTML, no `window.*` global exports, no mock `data.js`. Use real components, real router, real data fetching.
- **Theme:** dark-first; apply theme/accent/density live by swapping CSS variables on `:root`/`body`.
